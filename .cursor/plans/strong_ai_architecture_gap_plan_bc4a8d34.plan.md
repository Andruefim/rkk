# План выравнивания RKK под принципы сильного ИИ (снизу вверх)

## Граф зависимостей между фазами

Линейный порядок A→B→… нарушал реальные зависимости. Ниже — **обязательные отношения**:

- **G (veto / allostasis) до или вместе с F (efference)** — нельзя замыкать «WM предсказывает падение → команда» без veto на критические состояния.
- **E (s/z) блокирует осмысленную I (контекст)** — быстрый inner-loop или posterior по контексту требует явного **z**; Phase I не начинается до стабилизации E под флагом.
- **B (precision) — частичная зависимость для полной C** — маршрутизация рефлекс vs планирование по «уверенности канала» опирается на precision по модальностям; **полная** фаза C идёт после B (и после того, как есть шкала confidence).

**Рекомендуемый порядок внедрения:**

```
A → B → C(частично) → G(veto) → D → F → E → C(полная) → I → H
```

Смысл: сначала семантика Pearl и шкала ошибок (A, B), минимальные частотные ограничения (C₁), **безопасность** (G), память и связка sleep→WM (D), замыкание моторной петли с veto (F), затем разделение латента (E), полная временная иерархия с precision (C₂), контекстный posterior (I), язык/ToM (H).

## Поперечная дисциплина: флаги, baseline, регрессии

Вводится **на каждую фазу**, не только для FE:

- **Env-флаг на фазу** — новый путь включается явно (`RKK_PEARL_API=1`, `RKK_PRECISION_GROUPS=1`, `RKK_VALUE_VETO=1`, `RKK_SZ_SPLIT=1`, …); старый код остаётся до стабилизации.
- **Baseline до старта работ** — зафиксировать в журнале / скрипте: средний агрегированный PE за окно, средний `posture_stability`, среднее время `tick_step`, счётчик падений за N тиков, при наличии — intrinsic curve.
- **После каждой фазы — smoke**: минимум «500 engine-тиков без необработанного исключения» и порог падений не хуже baseline ± допуск (задать численно в CI или ручном чеклисте).
- **`RKK_FE_DECOMP=1`** остаётся отдельным флагом для разложения свободной энергии, не смешивать с обязательными флагами фаз.

| Принцип | Где в коде | Разрыв |
|---------|------------|--------|

## Фазы с измеримыми критериями завершения (acceptance)

| Фаза | Критерий готовности (измеримо) |
|------|--------------------------------|
| **A** | Юнит-тест: `counterfactual(X, a1, a2, anchors)` даёт результат, отличный от тривиального `intervene_predict(X, a1)` при том же X при подобранных `a1≠a2`; отдельный тест: при фиксированных **экзогенных** узлах (список U) контрфакт не меняет U. Документирован список узлов U в коде. |
| **B** | После калибровочного прогона: при инъекции шума в зрительный канал **precision vision** (или эквивалентный скаляр группы) снижается ≥2× за **N** шагов без ручной подстройки; **N = 50** по умолчанию (настраивается через `RKK_PRECISION_CALIB_WINDOW`). PE по каналам взвешивается precision × ошибка² в одном агрегаторе под флагом. |
| **C₁** | Рефлекторный путь не вызывает imagination/LLM (проверка вызовов или лог-ассерт в dev); ноги под CPG не получают прямых conflicting `do` из EIG (существующий блок усилен документацией). |
| **G** | Интеграционный тест: при injected critical state (напр. `com_z` ниже порога / флаг «fall imminent») veto срабатывает **100%**, независимо от весов прочих целей; простой прогноз энергии изменяет приоритет до критического порога. |
| **D** | Буфер/дамп: у каждого сохраняемого эпизода непустое поле **`physics_context`** (согласованная схема с `get_dynamics_params()`); топ-10% по surprise содержат ≥80% аномальных событий по разметке; REM вызывает **`wm.train_step`** на батче из `top_k` хотя бы раз за цикл сна под флагом. |
| **F** | В стационарных условиях корреляция `expected_obs` vs `actual_obs` **> 0.9** по **стационарным каналам** (proprio, joint torques и т.п. — список в конфиге); для **vision** — отдельный, более мягкий порог (например **> 0.7**), зафиксированный в конфиге (`RKK_EFFERENCE_CORR_PROPRIO`, `RKK_EFFERENCE_CORR_VISION` или один документированный блок). При инъекции аномалии среды дельта превышает порог и триггерит precision-down или запись в episodic; **команда опасного класса не уходит мимо veto** (совместно с G). |
| **E** | Под `RKK_SZ_SPLIT=1`: две головы, откат флагом в исходное поведение без потери основных весов ядра; первые 1000 шагов только обучение **z-головы** (s заморожена); метрика: ни один из подпространств не схлопывается в ноль (норма градиента z и s в допусках). |
| **C₂** | Решение «рефлекс vs план» использует групповые precision (из B); тест: при завышенной uncertainty зрения приоритет смещается к проприо в агрегаторе PE (измеримо на синтетике). |
| **I** | Вместо абстрактного MAML: реализован **PEARL-подобный** апдейт — posterior z из последних K наблюдений (или amortized сеть), без bi-level SGD на каждом выбросе; стабильность: скользящее среднее PE после смены «режима физики» не хуже baseline+ε за M шагов на валидационном сценарии. **Prerequisite:** эпизоды из Phase D содержат **`physics_context`** — иначе outer-loop / task label для posterior не отличит «другая физика» от «обычный шум». |
| **H** | Отложено: критерии задаются отдельным milestone. |

## Фаза A — расширение (Pearl + экзогенные переменные)

Помимо обёрток L1/L2/L3:

- В [causal_graph.py](backend/engine/causal_graph.py) (или конфиг рядом с VAR_NAMES): явная разметка **`EXOGENOUS_NODE_IDS`** — шум среды, начальные условия, параметры, не задаваемые политикой агента.
- Семантика **L3**: контрфакт = фиксация **U** (экзогены), интервенция только по контрфактному действию в эндогенных; не «весь X заморожен», чтобы избежать псевдо-L3 и смешения уровней (Simpson / неправильное условирование).
- Документация в docstring модуля API: отличие от условного rollout.

---

## Фаза B — precision + калибровочный протокол

1. **Оффлайн калибровка (разово, PyBullet):** N эпизодов с контролируемым Gaussian-шумом на одном канале (например vision slots, σ ∈ {0.1, 0.3, 0.5}).
2. Замер **PE по группам узлов** → эмпирический prior дисперсии канала → **инициализация precision-скаляров** (не «из воздуха»).
3. **Online:** дообучение precision (или лог-precision) градиентом по суррогату negative log-likelihood сенсорного предсказания, под флагом.

Без этого слой precision остаётся эвристикой; критерий завершения фазы B включает успешную калибровку по п.1–2. Окно **N** для acceptance (п. «precision vision ≥2×»): **`RKK_PRECISION_CALIB_WINDOW`** по умолчанию **50** шагов.

## Фаза C — два этапа

- **C₁ (после B, до полной confidence-маршрутизации):** документированные запреты и разделение путей в [mixin_tick.py](backend/engine/features/simulation/mixin_tick.py) / CPG-пайплайне.
- **C₂ (после E и B):** полная маршрутизация «рефлекс vs план» с использованием **group precision** из B.

---

## Фаза G — veto и аллостаз (перед / вместе с F)

- [value_layer.py](backend/engine/value_layer.py): лексикографический **veto** для физической целостности и критических состояний.
- Простой **прогноз расхода энергии** (скользящее среднее + признак задачи) для раннего «иди к зарядке».

Вместе с F: efference-путь проверяет WM-ожидание → veto может отменить или клипать команду до шага среды.

---

## Фаза D — память, surprise, Sleep → WM

- Отбор эпизодов по агрегированному **PE / surprise** из WM и графа.
- **Метаданные физики (prerequisite для Phase I):** каждый сохраняемый эпизод включает поле **`physics_context`** — словарь параметров среды на момент записи (фрикция, гравитация, масса/масштаб тела и т.д., по возможности из **`env.get_dynamics_params()`** или эквивалента после добавления в среду). Без этого поля posterior по **z** в Phase I не может различить «иная физика» и «обычный шум» при outer-loop / разметке задач.
- В **REM** ([sleep_consolidation.py](backend/engine/sleep_consolidation.py)): явный цикл  
  `top_episodes = episodic_memory.get_top_k_by_surprise(k)` → для каждого эпизода (или батчем) **`wm.train_step(obs_sequence, action_sequence)`** на ядре [temporal_world_model.py](backend/engine/temporal_world_model.py) / `CausalGraph` — не только replay политики/MC.

Критерий готовности — в таблице acceptance (D).

## Фаза F — efference copy

- Перед крупным изменением intent: короткий прогон WM → `expected_obs`.
- После шага среды: сравнение с `actual_obs`, вклад в PE и обучение.
- Интеграция с G: опасное `expected_obs` → veto.

---

## Фаза E — двухголовая s/z (без разрезания размерности ядра)

Риск разбиения `d` на две части: коллапс одного подпространства. Рекомендация в плане:

- **Добавить вторую голову** — линейный проектор поверх существующего эмбеддинга узлов → вектор **z**; **s** остаётся текущим представлением переходов без разрушения основной матрицы весов.
- Флаг **`RKK_SZ_SPLIT=1`**: при выключении — поведение как сейчас, основные веса ядра не меняются.
- **Первые 1000 шагов:** заморозка головы **s** / основного предсказателя переходов, обучение только **z**-головы и связанных контекстных членов лосса.

---

## Фаза I — PEARL-подобный контекст (вместо голого MAML-лайт)

- **Входные данные:** outer-loop и различение режимов опираются на эпизоды с **`physics_context`** из Phase D; без этого «исторические эпизоды с разной физикой» не имеют метки задачи — добавить контракт среды (`get_dynamics_params`) при необходимости раньше Phase I.
- Задача «смена физики» = смена режима в эмпирическом распределении эпизодов (task distribution задаётся явно; метка задачи — из `physics_context` и/или из признаков эпизода).
- **Не** чистый двухуровневый SGD по z на каждом выбросе.
- **PEARL-подход:** z как **posterior** из последних K шагов наблюдений (рекуррентный или amortized encoder); опционально outer-loop на исторических эпизодах с разной физикой без нестабильного inner SGD на шуме.
- Отдельные гиперпараметры только если останется градиентный fine-tune — задокументировать learning rates.

Критерий — строка в таблице acceptance для I.

## Риски (обновлённые)

- Неверная разметка экзогенных узлов в A приводит к ложным контрфактам — ревью списка U обязательно перед опорой на L3 в обучении.
- Калибровка B без логирования шумовых прогонов не воспроизводима — хранить seed и σ в артефакте прогона.
- REM→WM при завышенном lr может переобучить WM на артефакты симуляции — начать с малых lr и капа батча из top-k.
- Неполный или несопоставимый **`physics_context`** между записями эпизодов ломает обучение контекста в I — версионировать схему словаря при изменении PyBullet/URDF.
