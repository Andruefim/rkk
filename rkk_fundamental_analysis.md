# Фундаментальный анализ RKK Nova — Почему 2000 тиков не учат ходить

## Диагностика: Что говорят цифры

| Метрика | Значение | Что это значит |
|---------|----------|----------------|
| FALLEN × 1312 за 2054 тика | Падение каждые ~1.57 тика | Агент проводит >95% времени на земле |
| α trust = 3% | W_masked().abs().max() ≈ 0 | GNN **не выучила почти ничего** |
| Discovery = 0% | GT edges не обнаружены | Каузальная модель пуста |
| CG dG/dt = -2.5 | Compression ухудшается | Модель **деградирует**, не улучшается |
| block% = 1.7% | VL почти ничего не блокирует | Защита не работает (нечего защищать) |
| actual_ig ≈ 0.37 | Информационный выигрыш мал | Каждый do() мало чему учит |

> [!CAUTION]
> **Главный симптом**: После 1500 тиков fixed_root (торс закреплён) агент «отпускается» и немедленно падает. GNN за 1500 тиков закреплённого обучения не выучила ничего, что помогает стоять.

---

## Три уровня проблемы

### Уровень 1: Несовпадение гранулярности (Granularity Mismatch)

Это **главная** фундаментальная проблема, и она не техническая, а архитектурная.

**Текущая петля обучения:**
```
Каждый тик:
  1. score_interventions() → выбери пару (variable, value) с макс EIG
  2. do(variable=value) → один шаг физики
  3. observe() → заметь что изменилось
  4. actual_ig = |predicted - observed| → обучи System1
  5. train_step() → обучи GNN на (X_t, a_t) → X_{t+1}
```

**Проблема**: Ходьба — это **ритм длиной в сотни тиков**. Один `do(intent_stride=0.41)` за один тик не содержит почти никакой информации о том, работает ли ходьба. Это как пытаться выучить музыку, слыша по одной ноте в день без контекста мелодии.

GNN обучается предсказывать `X_{t+1}` из `(X_t, a_t)`. Но:
- `X_t` = 143 переменных (суставы, позы, слоты, интенты...)
- `a_t` = изменение одной переменной на одно значение
- `X_{t+1}` = состояние через 1 тик

С 2056 примерами в 143-мерном пространстве, GNN **математически не может** выучить нетривиальную динамику. Это не вопрос архитектуры — это curse of dimensionality.

### Уровень 2: Ловушка падения (Fall Trap)

```
[IGDiag] is_fallen=True в подавляющем большинстве тиков
```

После отпускания fixed_root на тике 1500, агент падает практически мгновенно и остаётся лежать. `reset_stance()` поднимает его, но он снова падает. В этом состоянии:

- **Все наблюдения выглядят одинаково**: "я на полу". GNN не может различить состояния.
- **Все действия имеют одинаковый эффект**: "ничего не меняется, я на полу". IG → 0.
- **CPG генерирует паттерны ходьбы** для тела, которое лежит на полу — бессмысленно.
- **CausalMotorExecutor** читает W (которая ≈ 0) и не производит никаких команд (alpha < 0.25).

> [!IMPORTANT]
> **Порочный круг**: Агент лежит → все do() одинаковы → IG ≈ 0 → GNN не учится → агент не знает как встать → агент лежит.

### Уровень 3: Фаза fixed_root не учит стоять

1500 тиков fixed_root должны были научить GNN связям intent → joint. Но:

- **do() по slot_* и intent_*** в fixed_root — это рандомные шевеления рук/ног привязанного тела. Тело не падает, поэтому нет негативного сигнала. Нет причинного следствия "если двигаю ногой не так, теряю баланс".
- **Переход fixed_root → free** — разрыв распределения (distribution shift). Всё, что GNN выучила в fixed_root, **нерелевантно** для свободного баланса, потому что физика полностью другая.
- **LeWM sequence training** обучается на траекториях закреплённого тела. Эти траектории бесполезны для свободной стоянки.

## Почему существующие компоненты не решают проблему

| Компонент | Задумка | Почему не работает |
|-----------|---------|-------------------|
| **CEM Planner** | Model-based action selection через WM | WM пустая (alpha=3%), CEM оптимизирует случайный шум |
| **GoalImagination** | Генерировать цели через GNN rollout | rollout через пустую GNN = рандом |
| **CPG** | Ритм ходьбы | Работает! Но GNN не может взять кредит за CPG |
| **CausalMotorExecutor** | intent × W → суставы | W ≈ 0, alpha < 0.25 → вывод отключён |
| **EIG scoring** | Выбрать максимально информативное do() | В 143D все do() одинаково не-информативны |
| **IntrinsicObjective** | Compression gain как награда | Граф не улучшается → CG < 0 → нет учебного сигнала |
| **AMP reward** | Adversarial motion prior | MotionDiscriminator(8 dims) без предобучения → шум |
| **JEPA latent** | Предсказание в латентном пространстве | Cosine loss → anti-collapse, но энкодер не видит обучающих данных |

## Твоя гипотеза: правильная ли она?

> *«без большого количества данных интеллект не может стать умным»*

**Частично да, но не совсем.** Проблема не в объёме данных per se, а в **информативности** данных.

2000 рандомных do() в 143D → почти нулевая информация.
200 **целенаправленных** экспериментов в 6D (только ноги) → достаточно выучить стоять.

Человеческий младенец не делает миллион случайных движений. Он:
1. Много месяцев тренирует **только** контроль головы
2. Потом **только** переворачивание
3. Потом **только** ползание  
4. Потом встаёт с опорой
5. Потом один шаг

Каждый этап — **низкоразмерная** задача с **чётким** сигналом (упал/не упал).

> *«AGI гуманоид не может каузально обучаться данным, если не испытывает их на практике»*

**Абсолютно верно.** Это ключевой тезис embodied cognition. Видео-обучение может дать prior (как выглядит ходьба), но inverse model (как **мои** intent_* превращаются в ходьбу) может быть выучена только через practice.

> *«Возможно, решением было бы встраивание в open-world игру»*

Это **не решит** фундаментальную проблему. Minecraft или любая open-world игра имеет ту же проблему: если агент не умеет управлять своим телом, его просто завалит первый крипер. Больше разнообразия → ещё больше curse of dimensionality.

Open-world **мог бы** помочь после того, как базовое владение телом уже работает — для обучения высокоуровневому планированию, навигации, tool use. Но сначала нужно решить задачу «стоять → ходить».

## Что я думаю: коренная причина и направление решения

### Коренная причина

**Система пытается решить задачу управления (control) инструментами анализа (causal discovery).** 

Causal graph learning ≠ motor control learning. NOTEARS/GNN предназначены для обнаружения каузальной структуры в наблюдательных данных. Но задача «научиться ходить» — это задача policy optimization, а не structure learning.

Текущая архитектура спрашивает: «Какой **один** do() максимально **информативен** для **модели мира**?»

А нужно спрашивать: «Какая **последовательность** intent_* максимизирует **время стоянки/ходьбы**?»

### Предложение: что я бы исследовал (не бездумный рецепт, а направление)

Я не буду говорить «добавь PPO» или «используй SAC». Я вижу, что ты строишь нечто принципиально другое — систему, которая сама открывает каузальную структуру мира. Это благородная цель. Вот где я вижу зазор между целью и реализацией:

#### 1. Trajectory-Level Contrastive Causal Learning

Вместо обучения на одиночных переходах `(s_t, a_t) → s_{t+1}`, обучай на **эпизодах**:

```
Эпизод = последовательность (s_0, a_0, s_1, a_1, ..., s_N)
Метка  = {stayed_upright, fell_backward, fell_forward, walked_forward}
```

GNN должна учиться отличать **паттерны в траекториях**, которые приводят к разным исходам. Это не RL — это всё ещё каузальное обнаружение, но на уровне траекторий, а не переходов.

Контрастивный подход: «Чем отличаются успешные эпизоды (стоял 50 тиков) от неуспешных (упал через 3 тика)?» → GNN учит **инварианты устойчивости**, а не покомпонентную динамику.

У тебя уже есть `EpisodicMemory` — он собирает fall episodes. Но сейчас эти эпизоды только логгируются. Они должны стать **основным обучающим сигналом** для WM.

#### 2. Прогрессивная размерность (Progressive Dimensionality)

Вместо 143D с первого тика:

```
Phase 0 (tick 0–500):    управляй только 6 vars (ноги) + 4 obs (posture, com_z, feet)
Phase 1 (tick 500–1500): добавь торс (spine_pitch, spine_yaw)
Phase 2 (tick 1500+):    добавь руки, голову
Phase 3 (tick 3000+):    добавь slot_* (визуальное восприятие)
```

На каждой фазе: GNN размерности ≤ 12. Выучить 12D → реально за 200–500 тиков. Критерий перехода: posture_stability > 0.7 в течение 100 тиков подряд.

Это **не хардкод поведения** — это хардкод **учебной программы** (какие переменные доступны). Содержание поведения агент всё ещё открывает сам.

Твой `PhysicalCurriculum` идёт в верном направлении, но он управляет **intent_targets** (что делать), а не **видимыми переменными** (из чего учиться). Разница принципиальна.

#### 3. Фаза fixed_root должна учить через controlled perturbations

Сейчас fixed_root = торс жёстко закреплён, руки/ноги шевелятся рандомно.

**Вместо этого**: fixed_root с **контролируемыми возмущениями** (учить баланс до отпускания):

- Периодически толкать торс случайной силой (виртуально, через изменение com_z/com_x целей)  
- Наблюдать, какие intent_* + joint комбинации компенсируют толчок
- GNN учит: «при возмущении X компенсация Y восстанавливает posture»

Так к моменту отпускания GNN уже знает **reactive balance**, а не просто «как двигать руками».

#### 4. Temporal Abstraction в GNN

Текущая GNN предсказывает s_{t+1} из s_t. Это **слишком мелко** для ходьбы.

Добавь **multi-scale prediction heads**:
- Head 1: предсказание s_{t+1} (текущее, 1 шаг)
- Head 5: предсказание s_{t+5} (полшага CPG)
- Head 20: предсказание s_{t+20} (полный цикл ходьбы)

Отдельные head'ы — простые Linear поверх тех же latent embeddings. Loss = сумма MSE на всех масштабах. Это заставит GNN учить **не мгновенную динамику, а тренды** — что и нужно для ходьбы.

У тебя есть `MultiscaleTime` (τ0=3, τ1=30, τ2=300, τ3=3600) — но он используется для temporal blankets (EMA фильтрация), а не для prediction heads. Prediction heads на разных масштабах — это другое и более мощное.

## Чего я НЕ предлагаю

- ❌ **PPO/SAC/стандартное RL** — ты строишь другую парадигму, RL rewards не нужны
- ❌ **Больше видео** — inverse model не учится из видео, ты прав
- ❌ **Больше модулей** — у тебя уже 66+ файлов в engine/, сложность — враг
- ❌ **Open-world** — преждевременно, сначала базовая моторика
- ❌ **Больше тиков** — 2000 vs 20000 не поможет если сигнал = 0 в каждом тике
- ❌ **Фикс конкретного бага** — как ты верно заметил, это не баг

## Итог: мой диагноз

**Система архитектурно не способна выучить ходьбу в её текущей конфигурации**, потому что:

1. **Гранулярность обучения** (один do() за тик) не совпадает с гранулярностью задачи (ритмическая координация сотен тиков)
2. **Размерность** (143D) слишком велика для объёма данных (2000 примеров)  
3. **Fall trap** делает 95% обучающих данных uninformative
4. **Fixed_root → free** — distribution shift, обнуляющий всё выученное

Самый перспективный путь, на мой взгляд:
1. **Резко снизить размерность** (progressive variable access)
2. **Учить на уровне эпизодов** (contrastive trajectory learning), а не переходов
3. **Controlled perturbations** вместо пассивного fixed_root
4. **Multi-scale prediction** чтобы GNN видела ходьбу как целое

Каждый из этих пунктов — **фундаментальное изменение архитектуры**, не фикс и не костыль. Они требуют значительного рефакторинга. Но они нацелены точно на те проблемы, которые я вижу в логах.
